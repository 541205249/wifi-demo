# 编码表方案评估

## 1. 先说结论

你最早定的这套“固定编号 + 外部 CSV 填真实指令”的方案，**现在依然是可行的，而且对你当前阶段是合适的**。

如果按你项目现在的阶段来判断，我的建议是：

- **短中期继续保留 CSV 方案**
- **不要现在就推翻重做**
- 但要把它定位成“人维护的协议映射层”，而不是长期唯一配置中心

也就是说，这套方案适合你现在“协议未完全定、App 要先把流程搭起来”的阶段；等后面协议稳定、团队协作复杂度上来，再考虑升级。

## 2. 这套 CSV 方案为什么现在能行得通

它最大的价值，不是“技术上多高级”，而是它非常适合你当前的协作方式。

### 2.1 它把 App 编码和设备真实命令解耦了

现在 App 只依赖：

- `s120101`
- `r120202`

而不直接依赖：

- `AT+EXAM=START`
- `RESULT+SPH=...`

这意味着：

- 设备端改真实命令时，App 编号不需要跟着改
- App 可以先把流程和页面都按编号写完
- 联调时优先改表，不急着改代码

### 2.2 它非常适合设备端同事维护

对设备端开发人员来说，CSV 有几个现实优势：

- 可以直接用 Excel / WPS 打开
- 学习成本低
- 修改成本低
- 可以通过 IM、邮件、网盘直接发文件

这比一上来要求对方去维护 JSON、YAML、Proto，协作门槛低很多。

### 2.3 它很适合版本对比

CSV 是纯文本：

- Git diff 直接可看
- 哪个编码变了很直观
- 哪条指令新增 / 删除一眼能看出来

对联调期来说，这比二进制 `xlsx` 稳很多。

## 3. 这套方案的短板也很明确

它不是没有问题，只是这些问题在你当前阶段还没大到必须推翻它。

### 3.1 表达能力有限

现在你定义的列只有：

- 编码
- 指令
- 编号解释

这已经足够轻量，但也意味着它不适合承载太复杂的规则，比如：

- 指令版本
- 超时设置
- 重试次数
- 是否必须 ACK
- 参数类型约束
- 是否仅某机型可用

这些信息如果以后越来越多，CSV 会开始吃力。

### 3.2 容易出现人工填写错误

比如：

- 编码写错
- 指令少一个符号
- 正则写坏
- 前缀拼错

虽然现在已经有校验机制，但 CSV 本质上还是“人为编辑的文本”，不能完全避免这类问题。

### 3.3 不太适合超大规模协议

如果后面编码量从几十条涨到几百条、上千条，CSV 还可以用，但维护体验会明显下降：

- 查找困难
- 审核困难
- 模块差异管理困难
- 多版本兼容困难

## 4. 有没有更好的方案

有，但要分阶段看。

### 4.1 对你现在最好的方案

**现在最好的方案不是“换掉 CSV”，而是“继续用 CSV，但把边界定义清楚”。**

建议把 CSV 明确定位成：

- 给人维护的协议映射表
- 只维护“编码 -> 指令 / 匹配规则”
- 不承载复杂运行策略

而复杂规则放到代码或后续独立配置里。

这其实是当前阶段性价比最高的做法。

### 4.2 比 CSV 更强的一种升级方案

如果后面你觉得 CSV 不够用了，我更推荐的升级方式不是直接切到 `xlsx`，而是：

**CSV 给人维护，App 内部再转成结构化模型。**

可以理解成两层：

- 第一层：CSV，设备端和产品都容易维护
- 第二层：App 读取 CSV 后，转换成内部 `CommandDefinition` / `ProtocolRule` 模型

你现在实际上已经在这么做了，只不过内部模型还比较轻。

以后如果需要升级，可以继续往这个方向走，而不是推翻。

### 4.3 再往后的一种更强方案

如果未来协议非常稳定、团队也更规范，可以升级成：

- `commands.csv`
- `command_meta.json`

或者：

- 一个完整的 `commands.json`

其中：

- `CSV` 继续给人看
- `JSON` 用来承载机器更关心的规则

例如 JSON 里可以放：

- `timeoutMs`
- `retryCount`
- `needAck`
- `ackRef`
- `payloadType`
- `module`
- `deviceModel`
- `minProtocolVersion`

这会比纯 CSV 更强。

## 5. 我最不推荐的方案

### 5.1 直接改成 xlsx

不推荐原因：

- 手机端解析成本高
- 调试麻烦
- Git diff 不友好
- 其实并没有解决 CSV 的核心短板

### 5.2 直接把真实指令重新写死到代码里

也不推荐，因为这会把你现在已经解耦好的结构重新耦合回去：

- 设备一改协议，App 就要发版
- 联调时改动会变重
- 设备端和 App 端很容易不同步

### 5.3 现在就上很重的 DSL / 协议平台

现在阶段也没必要。

因为你现在最稀缺的不是“超复杂协议建模能力”，而是：

- 快速把业务流程搭起来
- 让设备端能低成本参与
- 尽快完成联调

## 6. 我对你这个项目的建议

我建议你按下面三阶段走。

### 阶段一：现在

继续使用当前三列表 CSV：

- 编码
- 指令
- 编号解释

这个阶段重点是：

- 让真实模块先跑通
- 让业务页面都先按编号接起来
- 把联调闭环跑通

### 阶段二：协议开始稳定后

开始补“协议元数据”，但先不要破坏当前 CSV。

推荐新增一份轻量元数据配置，比如：

- `command_meta.json`

它只给 App 用，不要求设备端维护。

### 阶段三：协议规模变大后

如果以后真的有很多设备型号、很多协议版本、很多指令分支，再考虑：

- 配置中心
- 远程下发命令配置
- 多 profile / 多版本协议管理

这时候再做更重的体系，收益才够大。

## 7. 最终结论

你的“编号 + 指令 CSV”方案：

- **不是临时凑合方案**
- **现在依然成立**
- **对当前阶段是正确选择**

更准确地说，它现在已经不是问题本身，真正要补的是：

- 真实协议联调
- ACK / 超时 / 重试规则
- 协议版本管理
- 正式业务状态机

所以我的建议是：

- 现在不换方案
- 继续把 CSV 方案用到底
- 等协议稳定后，再在它上面加一层元数据配置

如果只给一句最短建议，就是：

**现在继续用 CSV，后面升级成“CSV + 元数据”会比现在直接推翻重做更合适。**
