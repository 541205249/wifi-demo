# 代码整洁检查清单

## 目录

- 基础阅读项：命名、函数、类、注释、错误处理、重复代码
- 页面类判断重点：工作台/容器页、设置表单页、历史详情页、业务台账页、程序页、报告页、设备连接页、目录入口页、按钮式集合页、Tab 场景页
- 高风险边界提醒：协议/网络辅助类、前台 Service、并发通信类、事务型 Store、配置仓库、权限委托、日志导出与日志持久化相关类
- 一般原则：优先保护显式业务清单、字段顺序、时序语义和低副作用 helper 提炼

## 命名

- 名称必须表达意图，避免 `data`、`info`、`manager`、`util` 这类过泛名字，除非上下文已经足够具体。
- 布尔值使用可判断真假的命名。
- 集合使用复数或集合语义命名。
- 如果重命名不能明显降低理解成本，就不要动。
- 对协议键、状态键、动作键这类跨方法复用的字符串，优先提炼成常量，避免同一语义在多个条件链中漂移。

## 函数

- 一个函数优先只做一件完整的事。
- 如果一个函数同时承担解析、校验、状态修改、IO 或 UI 更新，优先考虑拆分。
- 参数过多时，先判断是否缺少上下文对象或职责切分，而不是直接继续加参数。
- 避免通过长注释解释流程，优先通过提炼函数表达流程。
- 对协议或网络辅助类，优先把“响应解析”做成纯函数或近似纯函数，再决定是否调整重试、超时和传输流程。
- 对领域流程引擎或状态机，如果 `if/switch` 直接承载步骤语义、字段语义或跳转规则，先保持显式分支，不要急着改成配置表或规则引擎。
- 对 CSV/协议/配置加载器这类纯解析基础类，优先把 alias 字典、前缀规则、归一化规则收敛成常量或纯 helper，避免每次调用重复构造同一份映射。
- 对 demo / 教学型页面或 ViewModel，如果一段代码本来就在讲“这个场景怎么工作”，优先保留显式叙事和步骤清单；只提炼公共收尾、错误展示和状态落盘骨架。
- 对 seed data / 初始化样例类，如果代码本来就在陈列“默认病人、默认流程、默认图表、默认处方素材”，优先保留显式业务素材；不要为了消除长列表，把种子内容抽成难读的配置拼装。
- 对工作台/容器型页面，如果代码本来就在组织“哪一块 UI 显示什么、哪一组按钮控制什么”，优先保留显式页面骨架；只提炼稳定 key、重复动作壳和小型 UI helper。
- 对 Activity 级入口页，如果它承担文档选择、服务绑定、生命周期解绑和页面级按钮接线，优先保留显式入口流程；只提炼回调桥接、常量和纯目标解析 helper。
- 对传统 Demo Activity 或直接持有多个 View 的入口页，优先保留“初始化控件 -> 绑定服务 -> 接线按钮 -> 销毁清理”的线性流程；只提炼服务回调桥接、选中项解析和稳定提示文案 helper。
- 对设置表单页或双向字段映射页，如果代码本来就在显式表达“每个表单控件对应哪个业务字段”，优先保留 `bindXxx(...)` / `readXxxFromForm()` 的显式字段清单；只提炼文本读取、数字解析和格式化 helper。
- 对权限委托、系统授权闸门或 BaseUI delegate，如果它同时处理“弹提示 -> 发起申请 -> 接收结果 -> 可选跳设置页”，优先保留这条时序骨架；只提炼 Activity/Fragment 共用接线和纯结果收集 helper。
- 对日志设置页、导出设置页或运维工具页，如果页面本来就在显式陈列“本地导出 / 平台分享 / 应用配置 / 恢复默认”这些入口，优先保留显式功能清单；只提炼统一回调桥接和状态发布 helper。
- 对轻量异步场景 `ViewModel`，如果它通过多个 `runXxxExample()` 或 `executeXxx()` 入口触发演示流程，优先保留显式场景入口；只提炼运行中状态和结果发布 helper。
- 对演示功能面板 `Fragment` 或交互实验区页面，如果它本来就在显式陈列“刷新 / 新增 / 保存备注 / 打开弹框 / 请求权限”这类动作入口，优先保留显式按钮接线；只提炼输入读取、列表项构造和权限回调 helper。
- 对 demo 目录首页、说明入口页或样例导航页，如果页面本来就在列出演示模块入口，优先保留显式导航清单；只提炼统一跳转 helper 和页面 chrome 初始化 helper。
- 对 Fragment 容器型 Activity，如果它主要承担标题区、少量页面级按钮、Fragment 挂载和顶部状态镜像，优先保留显式容器骨架；只提炼 fragment 挂载 helper 和小型 render helper。
- 对按钮式场景集合页，如果页面本来就在显式陈列多个业务场景按钮，优先保留显式场景入口清单；只提炼页面 chrome、状态渲染和颜色切换 helper。
- 对 Tab + ViewPager 场景集合页，如果页面本来就在显式陈列分页标题和场景分组，优先保留显式 tab 清单与 pager 绑定；只提炼页面 chrome 和 tab 挂载 helper。
- 对历史详情页、设备日志页或带筛选的记录页，如果页面本来就在显式表达“摘要 + 过滤条件 + 日志正文”，优先保留显式过滤规则和摘要结构；只提炼页面 chrome、空态渲染和日志渲染壳 helper。
- 对业务台账列表页、档案页或“列表 + 弹窗编辑器”页面，如果页面本来就在显式表达“搜索 / 列表卡片 / 选中 / 编辑 / 导入”，优先保留显式动作清单和字段清单；只提炼卡片构造、当前选中摘要和保存壳 helper。
- 对程序页、步骤清单页或“当前程序 + 程序卡片 + 步骤卡片 + 自定义步骤弹窗”页面，如果页面本来就在显式表达业务流程顺序，优先保留 `renderHeader()`、`renderPrograms()`、`renderSteps()` 这类显式骨架；只提炼卡片构造、当前摘要和弹窗保存壳 helper。
- 对报告汇总页、指标清单页或“当前报告摘要 + 指标卡片 + 历史报告列表 + 二维码入口”页面，如果页面本来就在显式表达报告结构，优先保留 `renderCurrentReport()`、`renderMetrics()`、`renderHistory()` 这类显式骨架；只提炼摘要拼装、卡片构造、空态和轻量动作壳 helper。
- 对设备连接页、状态面板页或“服务状态 + 在线模块卡片 + 已知设备台账 + 日志面板”页面，如果页面本来就在显式表达运维操作入口，优先保留 `renderState()`、`renderConnectedDevices()`、`renderKnownDevices()`、`renderLogs()` 这类显式骨架；只提炼消息校验、卡片构造、按钮参数和日志拼装 helper。
- 对命令码展示 helper、长按提示或默认说明拼装这类微型辅助函数，只有在纯文本拼装重复时才提炼 helper；命令码常量清单本身继续显式保留。

## 类

- 如果一个类同时处理 UI、业务规则、存储、网络、格式化等多类责任，视为职责混杂。
- 优先按变化原因拆分类，而不是按字段数量机械拆分。
- 新增能力时，先判断应放到现有类、协作类还是更稳定的边界层。
- 对 `Activity`、`Fragment`、Controller 这类入口协调类，允许保留少量编排逻辑，但不应同时承担权限申请、导航路由、协议适配、网络细节和状态拼装。
- 如果入口类中出现多个“回调里做日志 + 拼文案 + 刷状态”的重复片段，优先考虑提炼统一的状态同步或事件分发辅助函数。
- 不要仅凭“公开方法很多”就判定一个类必须拆分。先区分这些方法是薄转发、状态入口，还是已经在类内做了解析、规则判断和副作用编排。
- `ViewModel` 可以作为界面状态门面存在，但如果它开始直接解析外部载荷、拼装领域对象或协调多套基础设施，应优先考虑提炼解析器、用例对象或协作服务。
- 协调层如果同时负责“决策”和“执行”，优先把稳定的决策映射或收尾动作提炼成辅助函数，让主流程只保留可读的编排骨架。
- 对前台 `Service`、socket、锁资源和监听器通知混在一起的类，优先视为受保护边界；先提炼旁路辅助函数，不要先打散主生命周期流程。
- 对“注册处理器/注册路由/注册编码映射”这类接线型 `ViewModel` 或协调类，优先保留显式注册清单；只提炼稳定的注册骨架和公共状态刷新 helper，不要急着外提成配置表。
- 对底层 dispatcher / router，如果它直接承载 payload 类型分发、session 生命周期和自动移除语义，先把它视为受保护核心；不要在一次补丁里同时改路由规则、异常语义和清理策略。
- 对导出、打包、分享、文件提供器这类日志辅助类，优先把它们当成资源边界；先提炼命名、过滤和纯格式化 helper，不要轻易重排分享、目录选择、文件删除和回调通知顺序。
- 对命令目录、Profile 和预留编码清单这类配置性结构，优先保留显式条目列表；只提炼默认说明字符串、纯命名 helper 或不带语义损失的构造骨架。
- 对命令码常量类、预留编号类或协议编码清单，优先保留显式常量/条目列表；不要为了“更面向对象”把稳定编码改成枚举树、DSL 或动态注册。
- 对纯格式化/数据装配类，如果它已经足够小且无副作用，先判断是否“已经足够好”；不要为了再拆一层 helper 反而让格式规则分散。
- 对设置页、配置页或表单型 `Fragment`，不要只因为字段多就急着做反射绑定、配置表驱动或通用表单引擎；先判断显式字段映射是不是页面可读性的一部分。
- 对权限代理这类基础委托，不要为了“统一接口”同时改动对话框文案、权限申请入口、结果回调和跳系统设置语义；优先每次只收敛一个稳定骨架。
- 对配置仓库、设置仓库或运行时配置门面，如果它同时处理 SharedPreferences、配置对象构建、运行时 `init/apply` 和摘要文案，优先保护落盘与应用时序；只提炼纯校验、纯格式化和纯发布 helper。
- 对薄网关接口、桥接接口或 service facade，如果它只是直接暴露少量稳定能力给上层使用，优先保持接口直接清晰；不要为了“更面向对象”再套一层无收益抽象。
- 对基础页面基类或 UI helper 基类，如果它主要提供共享 ViewModel 绑定、通用 UI helper 和少量页面壳能力，优先保持轻量直接；不要急着把小 helper 再拆成多层 delegate。
- 对轻量摘要模型、列表项模型或简单数据载体，如果类本身只承载字段和 getter，优先保持直接数据结构；不要为了“更规范”强行引入 builder、mapper 或额外包装。

## 注释

- 不要用注释掩盖含义不清的命名或混乱流程。
- 注释更适合解释业务背景、协议约束、兼容性原因和无法从代码直接看出的事实。
- 如果注释只是逐句翻译代码，优先删除或改成更清晰的代码结构。

## 错误处理

- 优先让错误处理路径清晰，而不是把异常吞掉。
- 如果必须降级或忽略异常，至少保留可观测信息或明确注释原因。
- 错误分支不要和主流程深度交织，能前置返回就前置返回。
- 对并发通信类里的异常处理，如果当前行为依赖“清理后再通知”或“仅在特定条件下通知”，不要把日志去重和异常路径重排放在同一次改动里。

## 重复代码

- 遇到重复逻辑时，先判断重复的是流程、算法还是偶然相似的语句。
- 只有在重复模式稳定、抽象不会增加理解成本时才提炼公共逻辑。
- 跨模块重复代码先判断是否有意分叉，不要默认强行合并。
- 如果多个方法都在重复“修改状态 -> 发布状态 -> 重建派生数据 -> 记录日志”，先确认这是不是稳定的状态变更模板，再决定是否提炼统一辅助方法。
- 提炼状态仓库模板时，优先抽“发布流程”和“收尾动作”，不要把具体业务分支、字段选择和边界条件一起塞进模板。
- 对持久化 `Store` / `Repository` 的事务内重复，优先提炼“准备实体”“插入日志”“端口/时间等格式归一化”这类稳定骨架；连接状态、计数规则、日志类别等业务差异继续显式保留。
- 如果一段代码本质上是在按顺序显式列出业务条目、报告行或展示项，并且每一行都有自己的文案、阈值或格式规则，通常先保持显式写法更稳。
