# 坏味道与重构动作

## 目录

- 通用坏味道：长函数、过大的类、入口类职责膨胀、协调层命令入口重复、重复代码、冗长条件分支、临时变量过多、功能修改与重构混在一起
- 高风险边界与基础设施：协议解析与传输耦合、前台 Service 保护边界、注册清单型接线类、纯解析器与字典型基础类、底层分发核心、日志导出与资源打包辅助类、日志持久化与崩溃监控辅助类、并发通信类重构
- 页面与交互模式：工作台 / 容器型页面、Activity 级入口页、传统 Demo Activity、设置表单页、权限委托、日志设置页、演示功能面板、Demo 目录首页、Fragment 容器型 Activity、按钮式场景集合页、Tab + ViewPager 场景集合页、历史详情页、业务台账页、程序页、报告页、设备连接页
- 业务数据与其他判断：Seed Data / 初始化素材类、配置仓库 / 运行时应用边界、轻量异步场景 Runner ViewModel、纯格式化 / 数据装配类、显式命令码常量清单与微型展示 helper、状态仓库方法过多、持久化仓库事务模板、显式状态机/流程引擎、行式业务清单、验证原则

## 长函数

现象：
- 一个函数同时处理多个阶段
- 需要通过长注释分段说明流程

优先动作：
- 提炼函数
- 引入解释性变量
- 把条件判断拆成意图明确的小函数

## 过大的类

现象：
- 一个类同时管理状态、流程控制、IO、格式化或外部交互

优先动作：
- 按职责拆分类
- 保持原有对外接口尽量稳定
- 先拆最独立、最容易验证的部分

## 入口类职责膨胀

现象：
- `Activity`、`Fragment` 或 Controller 同时负责页面初始化、权限处理、导航分发、服务绑定、协议回调和数据组装
- 类本身并不复杂，但“什么都沾一点”，导致后续每加一个需求都继续堆在入口类

优先动作：
- 保留入口类的编排职责，不要一上来把所有逻辑挪空
- 先提炼最稳定的辅助能力，例如路由解析、服务连接协调、事件日志拼装
- 优先消除重复回调处理，再考虑拆出独立协调器或委托类
- 如果涉及 Android 生命周期，先保持原有生命周期钩子和对外行为不变

## 协调层命令入口重复

现象：
- 多个命令方法都在执行“前置检查 -> 调协作对象 -> 追加日志或提示 -> 刷新状态”
- 业务动作不同，但收尾动作高度一致

优先动作：
- 保留命令入口的显式可读性
- 先提炼网关可用性检查、统一收尾动作或公共日志拼装
- 不要为了减少几行代码把不同命令折叠成难读的通用执行器

## 重复代码

现象：
- 相同流程在多个分支或多个类中重复出现
- 只有少量差异点

优先动作：
- 提炼公共流程
- 参数化差异点
- 如果重复跨模块，先确认是否真的应该收敛到公共层

## 冗长条件分支

现象：
- 多层 `if/else` 嵌套
- 分支条件难以一眼看懂

优先动作：
- 提炼判定函数
- 用守卫子句提前返回
- 把分支的意图命名出来

何时先不要重构成表驱动：
- 各分支的步长、边界、符号规则或副作用明显不一致
- 分支之间只是表面相似，实际业务语义不同

更稳的第一步：
- 先提炼跨方法复用的 key 常量
- 先把共同的发布流程或日志流程抽出来
- 等规则真正趋同后，再考虑表驱动

## 协议解析与传输耦合

现象：
- 一个网络辅助类同时处理重试、超时、socket IO 和响应解析
- 解析逻辑里存在重复的正则匹配、归一化或 fallback 处理

优先动作：
- 先提炼纯解析辅助函数
- 保持原有重试次数、超时配置和传输时序不变
- 只有在解析边界稳定后，再考虑调整 IO 或重试策略

## 前台 Service 保护边界

现象：
- 一个 Android `Service` 同时处理前台通知、生命周期、socket 监听、锁资源、监听器回调和状态归档
- 主流程里包含启动、停止、清理、断连、异常通知等时序敏感路径

优先动作：
- 先把这类类标记为受保护边界
- 只从无副作用旁路逻辑下手，例如日志拼装、简单参数构造、纯判定函数、格式化
- 保持 `onCreate/onStartCommand/onDestroy`、accept 循环、资源释放顺序和监听器通知顺序不变

何时先不要大拆：
- 一次改动同时涉及线程池、socket 生命周期、前台通知和断连归档
- 抽象后会把“何时启动”“何时停止”“何时回放缓存”这些时序信息藏进多层跳转里

更稳的第一步：
- 提炼局部辅助函数
- 明确记录哪些部分是本轮刻意不动的受保护边界

## 注册清单型接线类

现象：
- 一个 `ViewModel`、页面协调类或配置页类里集中注册很多 handler、路由或协议处理器
- 每一项注册看起来结构很像，但文案、编码和场景语义各不相同

优先动作：
- 保留显式注册清单，让调用处能直接看出“当前页面关心哪些编码/路由”
- 只提炼稳定的接线骨架，例如 `registerXxxUseCase(...)` 辅助函数
- 对加载结果、快照回填等重复状态刷新，提炼统一 state-update helper

何时先不要改成表驱动或外部配置：
- 抽象后需要来回跳转配置表、lambda 和页面代码才能看懂页面职责
- 每一项虽然结构相似，但说明文案、日志语义和场景含义并不完全一致

更稳的第一步：
- 让“注册哪些项”继续留在主函数里
- 只把重复的 lambda 外壳和 LiveData 刷新收敛起来

## 纯解析器与字典型基础类

现象：
- 一个加载器、解析器或格式转换器里频繁构造同一份 alias 字典、规则表或归一化映射
- 重复不在业务流程，而在纯数据字典或纯规则集合

优先动作：
- 把稳定字典提炼为静态常量或静态工厂
- 保持外围 IO、读取顺序、报错语义不变
- 继续让解析主流程保持线性、易读

何时先不要大改：
- 为了“更抽象”把本来直线型的加载流程拆成过多层级
- 在同一补丁里同时改 CSV 读取、字段回填、异常信息和规则字典

更稳的第一步：
- 先收敛 alias map、规则前缀、可复用的归一化辅助函数
- 让变更尽量停留在纯函数和纯数据层

## 展示型 Demo ViewModel

现象：
- 一个 demo 页面或教学型 ViewModel 里有很多场景方法，每个方法都在讲一个业务故事
- 各场景都要更新 UI 状态、控制台文本和成功/失败结果，但文案和步骤内容差异很大

优先动作：
- 保留每个场景方法的显式叙事和步骤清单
- 只提炼公共的 UI 状态落盘、失败收尾、控制台构建 helper
- 让“这个示例想表达什么”继续一眼可见

何时先不要做统一配置中心：
- 抽象后要跳到多层配置或模板里才能看懂每个示例场景
- 每个示例的文案、步骤和讲解重点本来就是页面价值的一部分

更稳的第一步：
- 把 `new UiState(...)`、错误提示和日志收尾收敛起来
- 场景脚本本身继续显式保留在方法里

## 底层分发核心

现象：
- 一个 dispatcher 或 router 同时负责 payload 分类、route key 解析、session 生命周期、fallback 和 auto-remove
- transfer / stream / ack / command 不只是分支不同，还带有不同的状态机和错误语义

优先动作：
- 先把它视为受保护核心
- 如果要动，优先只动日志、纯 key 构造、纯 fallback 上下文构造这类旁路能力
- 保持分发顺序、route key 语义、auto-remove 条件和异常返回结构不变

何时先不要大抽象：
- 为了减少几段相似分支，就把 command/ack/transfer/stream 混进同一个泛型模板
- 把“未命中处理器”和“处理器异常”的差异语义折叠掉

更稳的第一步：
- 先写进 skill 作为边界判断样本
- 等有明确重复且验证面足够时，再做更小颗粒度的 helper 提炼

## Seed Data / 初始化素材类

现象：
- 一个类集中创建默认病人、默认流程、默认图表、默认配置或示例会话
- 内容很多，看起来像长列表，但每一项都带业务语义、默认值和演示含义

优先动作：
- 保留显式素材清单，让业务同学和开发都能直接看到默认内容
- 只提炼真正重复的纯默认值、命名常量或无语义噪音的 helper
- 避免把种子内容折叠成难读的 builder 套 builder

何时先不要表驱动：
- 抽象后需要在多处跳转才能看懂默认流程到底长什么样
- 每条 seed 本身就是产品样例或业务素材，不只是普通配置数据

更稳的第一步：
- 把它写进 skill 作为“显式素材优先保留”的样本
- 只有在纯字段默认值明显重复时，再做局部 helper 提炼

## 日志导出与资源打包辅助类

现象：
- 一个辅助类同时处理目录选择、文件过滤、压缩、分享、缓存清理和主线程回调
- 流程里包含 `FileProvider`、`DocumentFile`、缓存目录删除、压缩结果分享等资源边界

优先动作：
- 先把它视为资源边界
- 只提炼文件名生成、日志过滤、时间格式化、纯文件遍历 helper
- 保持目录选择、分享 Intent、清理时机和回调通知顺序不变

何时先不要大改：
- 一次补丁同时改动分享链路、导出路径、缓存删除和过滤逻辑
- 为了“结构统一”打散原本线性的资源处理流程

更稳的第一步：
- 只从纯辅助格式化或纯遍历逻辑下手
- 把高风险资源时序先记录为 skill 里的保护边界

## 日志持久化与崩溃监控辅助类

现象：
- 一个日志收集类同时负责内存 buffer、spill、文件落盘、持久化恢复和 crash 监控
- 表面上存在格式化、命名、文件切分等重复，但主流程高度依赖时序

优先动作：
- 优先保护 flush、spill、持久化替换和 crash 处理时序
- 只提炼时间格式、文件名格式、纯大小计算和纯辅助函数
- 每次只动一个纯辅助点

何时先不要触碰主流程：
- 把持久化读写、异常恢复和 crash 处理混在同一补丁里调整
- 为了统一而改动 buffer 锁、文件锁或 flush 条件

更稳的第一步：
- 先抽时间格式和命名 helper
- 让改动停留在无副作用的纯辅助层

## 工作台 / 容器型页面

现象：
- 一个 `Fragment` 或页面类同时组织病人信息、流程进度、图表切换、按钮区和功能行
- 方法很多，但大部分都在描述页面结构和用户操作入口

优先动作：
- 保留显式页面骨架，让人一眼看出页面有哪些区域和交互
- 只提炼稳定的功能 key、重复 `OnClickListener` 外壳和小型 UI helper
- 让功能行、按钮组和观察关系继续显式陈列

何时先不要拆成多层配置：
- 抽象后得在多层 builder、配置表和 helper 之间来回跳转才能看懂页面
- 页面本身的价值就在于把工作台结构直接展开

更稳的第一步：
- 先把魔法字符串收成常量
- 先把重复动作壳提炼成小 helper

## 命令目录 / Profile 配置类

现象：
- 一个目录类或 Profile 类集中列出预留编码、模块名、动作名和默认说明
- 条目很多，但每一条都代表稳定协议位，不只是普通重复数据

优先动作：
- 保留显式条目清单
- 只提炼默认说明字符串、默认展示名或其他纯拼装 helper
- Profile 壳类保持简单直接，不为了“统一接口更漂亮”再加无收益层次

何时先不要抽成更深配置框架：
- 抽象后看不出实际有哪些预留编码
- 只是为了减少几次 `module/subModule/action` 拼接就引入额外 DSL

更稳的第一步：
- 让目录条目继续留在显式 builder 清单里
- 只把纯默认说明拼装抽成 helper

## Activity 级入口页

现象：
- 一个 Activity 同时处理文档选择器、服务绑定、生命周期解绑、按钮接线和服务回调桥接
- 代码不一定特别复杂，但入口流程和生命周期边界比较敏感

优先动作：
- 保留 `initWidgets()`、`loadData()`、`onDestroy()`、绑定服务和解绑服务的显式骨架
- 只提炼服务回调桥接 helper、MIME type 常量、选中目标解析 helper
- 保持启动服务、绑定服务、解绑服务和监听器清理顺序不变

何时先不要大拆：
- 为了“更干净”把页面级入口流程拆到多个 manager / delegate 中，导致生命周期难追踪
- 同一补丁里同时调整服务绑定方式、监听器注册方式和发送目标选择逻辑

更稳的第一步：
- 先消除回调里的重复 `runOnUiThread + trace + refresh`
- 先把稳定文档类型和广播标签提成常量

## 传统 Demo Activity / 直接持有 View 的入口页

现象：
- 页面直接持有 `TextView`、`Spinner`、`Button` 等 View，并承担服务绑定、按钮接线和日志展示
- 多个服务回调里重复 `post -> trace/log -> refresh`

优先动作：
- 保留入口页的线性操作流程，让“怎么演示、怎么操作”继续直接可读
- 把匿名服务监听器提成字段或小型桥接 helper
- 提炼选中 client/device 的解析 helper 和稳定文案常量

何时先不要大拆：
- 为了统一架构把页面拆到多个 delegate / manager，导致生命周期和 View 更新路径分散
- 同一补丁里同时改服务监听器注册、目标选择、UI 刷新和页面导航

更稳的第一步：
- 先收敛回调桥接和选中项校验
- 在 `onDestroy()`、`onServiceDisconnected()` 这类边界上显式做监听器清理

## 设置表单页 / 双向字段映射

现象：
- 一个 `Fragment` 或 Activity 同时包含 `bindSettings(...)`、`readSettingsFromForm()`、`bindViews()` 这类成对方法
- 字段很多，看起来冗长，但每一行都在表达“这个控件对应哪个设置项”

优先动作：
- 保留显式字段映射，让页面职责和字段含义直接可读
- 只提炼 `readText(...)`、`readDouble(...)`、`formatNumber(...)` 这类纯 helper
- 如果默认值来自稳定 seed data 或配置模型，优先统一默认值来源，而不是引入通用表单引擎

何时先不要做配置化：
- 为了减少重复，把字段映射改成反射、字段名字符串或配置表驱动
- 抽象后需要在控件 id、字段 key 和 lambda 之间来回跳转才能看懂页面

更稳的第一步：
- 先收敛文本读取、数字解析和格式化 helper
- 让 `bind...` / `read...` 的显式字段清单继续留在页面类里

## 权限委托 / 系统授权闸门

现象：
- 一个基础委托类同时处理权限提示弹框、Activity/Fragment 申请入口、`onRequestPermissionsResult(...)` 回调和跳系统设置页
- `Activity` 与 `Fragment` 版本流程几乎相同，只是最终请求动作不同

优先动作：
- 保留“检查权限 -> 可选弹框 -> 发起申请 -> 收集拒绝结果 -> 可选跳设置页”的时序骨架
- 只提炼共用申请骨架和拒绝权限收集 helper
- 保持 requestCode、callback 保存时机和设置页跳转语义不变

何时先不要大改：
- 为了“更抽象”把权限申请、结果处理和设置跳转拆到多个策略层
- 同一补丁里同时改文案默认值、回调语义、申请入口和拒绝后行为

更稳的第一步：
- 先收敛 Activity/Fragment 共用的 request helper
- 再提炼 denied list 收集这类纯结果辅助函数

## 日志设置页 / 导出桥接

现象：
- 一个设置页同时包含表单字段映射、应用配置按钮、恢复默认按钮，以及多个“本地导出 / 平台分享”入口
- 多个导出 hook 的真正重复点，不在功能入口本身，而在成功/失败回调桥接和状态发布

优先动作：
- 保留显式入口清单，让页面直接表达支持哪些导出动作
- 只提炼统一的导出结果 callback helper 和表单状态发布 helper
- 保持 hook 注册时机、按钮绑定关系和导出 API 调用顺序不变

何时先不要大拆：
- 为了“页面更干净”把导出入口、表单逻辑和状态观察拆到多个 manager / coordinator
- 同一补丁里同时改导出注册时机、五击触发规则、页面文案和配置表单结构

更稳的第一步：
- 先收敛 success/failure 回调桥接
- 再收敛 `formLiveData + summaryLiveData` 这类稳定发布骨架

## 配置仓库 / 运行时应用边界

现象：
- 一个 repository 同时处理 SharedPreferences 读写、配置对象构建、参数校验、运行时 `init/apply` 和 UI 摘要文案
- 看起来既像存储层，又像运行时门面，主流程带有明显副作用

优先动作：
- 先把它视为配置应用边界
- 只提炼纯校验、纯格式化、纯默认值 helper
- 保持“读配置 -> 校验 -> 构建 config -> 持久化 -> 应用到运行时”的顺序不变

何时先不要大拆：
- 为了分层整齐，把 SharedPreferences、Config Builder、运行时 `DLog.init(...)` 和 overlay 应用一次性拆散
- 同一补丁里同时改默认值来源、持久化 key、配置校验语义和运行时应用顺序

更稳的第一步：
- 先记录哪些部分属于运行时边界
- 先从摘要文案、数字格式化、参数校验 helper 下手

## 轻量异步场景 Runner ViewModel

现象：
- 一个 `ViewModel` 通过多个 `runXxxExample()` 或 `runXxxScenario()` 入口驱动异步请求

- 模板函数优先承接“取状态、发布状态、可选日志、可选派生刷新”，业务变更本身仍留在调用处

## 持久化仓库事务模板

现象：
- `Store` 或 `Repository` 中多个写方法都在做“取或建实体 -> 更新摘要 -> 修改业务字段 -> 存储 -> 写日志 -> 裁剪”
- 重复位于事务内骨架，而不是具体业务字段

优先动作：
- 保持事务边界和调用顺序不变
- 只提炼稳定骨架，例如准备实体、插入日志、端口归一化
- 让连接状态、计数规则、动作类型继续留在调用处显式表达

何时先不要抽成通用执行器：
- 为了统一两个写方法，引入过多回调接口、枚举分发或隐藏业务动作
- 抽象后调用处已经看不出这是“连接事件”还是“通信事件”

更稳的第一步：
- 先抽局部 helper，而不是一口气做高度参数化模板
- 如果 `app` 和 `demo` 有镜像实现，优先同步验证两边

## 显式状态机/流程引擎

现象：

- 先保护显式业务语义
- 只提炼真正重复的解析辅助函数或命名常量
- 保持步骤推进和跳步条件的可直接阅读性

何时先不要改成规则引擎或表驱动：
- 字段别名、比较符号、步进边界和日志语义还没有稳定模型
- 配置化后会让排查流程问题必须来回跳转数据表和解释器

更稳的第一步：
- 保留 `switch/if` 的显式结构
- 先把“为什么跳步”“依据哪个字段”写清楚在代码结构里，而不是藏进抽象层

## 行式业务清单

现象：
- 一个函数连续构造多条业务记录、展示项或报告行
- 每一条看起来结构相似，但文案、阈值、格式化规则和字段来源各不相同

何时先不要重构成表驱动：
- 为了消除表面重复而引入大量匿名配置、索引字段或映射表
- 抽象后反而让每一行的业务含义更难直接阅读

更稳的第一步：
- 保留显式行式写法
- 只提炼真正重复的格式化函数或公共计算
- 等条目结构和变化模式足够稳定，再考虑数据驱动

## 并发通信类重构

现象：
- 类同时处理线程调度、socket 生命周期、监听器通知、资源清理和日志

优先动作：
- 每次只改一个维度，优先做表层去重或提炼辅助函数
- 不要在同一补丁里同时改锁、断连通知、清理顺序和消息协议
- 对清理流程、断连通知和异常处理的改动，必须单独验证
- 优先从纯辅助路径下手，例如解码 fallback、日志拼装、格式化、无副作用判定函数

## 验证原则

- 每做完一轮小重构，就运行最小范围验证。
- 如果缺少自动化验证，至少说明人工验证路径。
- 无法验证的结构性调整，默认保守处理。
- 对入口类重构，至少覆盖页面打开、核心点击路径、服务绑定或关键回调是否仍然工作。
