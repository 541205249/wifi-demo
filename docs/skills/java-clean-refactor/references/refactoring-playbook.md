# 坏味道与重构动作

## 长函数

现象：
- 一个函数同时处理多个阶段
- 需要通过长注释分段说明流程

优先动作：
- 提炼函数
- 引入解释性变量
- 把条件判断拆成意图明确的小函数

## 过大的类

现象：
- 一个类同时管理状态、流程控制、IO、格式化或外部交互

优先动作：
- 按职责拆分类
- 保持原有对外接口尽量稳定
- 先拆最独立、最容易验证的部分

## 入口类职责膨胀

现象：
- `Activity`、`Fragment` 或 Controller 同时负责页面初始化、权限处理、导航分发、服务绑定、协议回调和数据组装
- 类本身并不复杂，但“什么都沾一点”，导致后续每加一个需求都继续堆在入口类

优先动作：
- 保留入口类的编排职责，不要一上来把所有逻辑挪空
- 先提炼最稳定的辅助能力，例如路由解析、服务连接协调、事件日志拼装
- 优先消除重复回调处理，再考虑拆出独立协调器或委托类
- 如果涉及 Android 生命周期，先保持原有生命周期钩子和对外行为不变

## 重复代码

现象：
- 相同流程在多个分支或多个类中重复出现
- 只有少量差异点

优先动作：
- 提炼公共流程
- 参数化差异点
- 如果重复跨模块，先确认是否真的应该收敛到公共层

## 冗长条件分支

现象：
- 多层 `if/else` 嵌套
- 分支条件难以一眼看懂

优先动作：
- 提炼判定函数
- 用守卫子句提前返回
- 把分支的意图命名出来

何时先不要重构成表驱动：
- 各分支的步长、边界、符号规则或副作用明显不一致
- 分支之间只是表面相似，实际业务语义不同

更稳的第一步：
- 先提炼跨方法复用的 key 常量
- 先把共同的发布流程或日志流程抽出来
- 等规则真正趋同后，再考虑表驱动

## 协议解析与传输耦合

现象：
- 一个网络辅助类同时处理重试、超时、socket IO 和响应解析
- 解析逻辑里存在重复的正则匹配、归一化或 fallback 处理

优先动作：
- 先提炼纯解析辅助函数
- 保持原有重试次数、超时配置和传输时序不变
- 只有在解析边界稳定后，再考虑调整 IO 或重试策略

## 临时变量过多

现象：
- 函数里存在大量只为拼接流程而出现的中间变量

优先动作：
- 提炼表达式
- 仅保留能提高可读性的中间变量
- 避免为了缩短代码而写成更难懂的链式逻辑

## 功能修改与重构混在一起

现象：
- 同一改动既在调整结构，又在改变外部行为

优先动作：
- 先拆开思考
- 能分批提交就分批
- 不能分批时，在说明里明确哪些部分是行为变化，哪些部分是纯重构

## 状态仓库方法过多

现象：
- Repository、Store 或 State Holder 暴露很多命令方法
- 多个方法都围绕同一份状态做小幅修改和派生数据刷新

优先动作：
- 先判断它是不是稳定的状态门面，而不是按方法数直接拆类
- 优先提炼重复的状态变更模板
- 如果解析、格式转换或跨边界协调逻辑开始混入，再考虑拆出协作对象
- 模板函数优先承接“取状态、发布状态、可选日志、可选派生刷新”，业务变更本身仍留在调用处

## 行式业务清单

现象：
- 一个函数连续构造多条业务记录、展示项或报告行
- 每一条看起来结构相似，但文案、阈值、格式化规则和字段来源各不相同

何时先不要重构成表驱动：
- 为了消除表面重复而引入大量匿名配置、索引字段或映射表
- 抽象后反而让每一行的业务含义更难直接阅读

更稳的第一步：
- 保留显式行式写法
- 只提炼真正重复的格式化函数或公共计算
- 等条目结构和变化模式足够稳定，再考虑数据驱动

## 并发通信类重构

现象：
- 类同时处理线程调度、socket 生命周期、监听器通知、资源清理和日志

优先动作：
- 每次只改一个维度，优先做表层去重或提炼辅助函数
- 不要在同一补丁里同时改锁、断连通知、清理顺序和消息协议
- 对清理流程、断连通知和异常处理的改动，必须单独验证
- 优先从纯辅助路径下手，例如解码 fallback、日志拼装、格式化、无副作用判定函数

## 验证原则

- 每做完一轮小重构，就运行最小范围验证。
- 如果缺少自动化验证，至少说明人工验证路径。
- 无法验证的结构性调整，默认保守处理。
- 对入口类重构，至少覆盖页面打开、核心点击路径、服务绑定或关键回调是否仍然工作。
