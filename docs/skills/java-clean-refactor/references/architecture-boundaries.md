# 边界与架构判断

## 模块落点

- 公共模块只放稳定、可复用、跨场景成立的能力，不放业务特例。
- 业务模块优先承载面向场景的流程和规则，不把偶发需求过早下沉为基础设施。
- 如果两个模块都出现相同基础能力，再判断是否值得下沉。

## 依赖方向

- UI 层不要直接依赖底层存储、通信或协议细节，优先通过清晰边界访问。
- 高层策略不应被低层实现细节反向牵制。
- 如果一个类知道太多下层细节，优先考虑收敛接口而不是继续扩展调用链。
- `ViewModel` 适合承接界面交互入口和状态暴露，但不应长期承担协议解析、字符串载荷拆解、持久化格式转换等可独立演化的细节。

## 抽象时机

- 出现一次重复，不急着抽象。
- 出现稳定的重复模式，再抽象。
- 如果抽象后只是多了一层转发而没有降低变化成本，先不抽。

## 兼容性与演进

- 对外协议、存储格式、硬件行为、历史数据兼容优先于结构整齐。
- 重构前先识别真实稳定点，不要在高风险边界上做样式型重写。
- 对线程、socket、硬件通信这类边界，优先保护时序和清理语义，再谈结构优化。
- 如果一段代码同时参与资源清理和监听器通知，把它视为受保护边界；优先在旁路辅助函数上做重构。

## 决策顺序

1. 先确认外部行为和兼容性约束。
2. 再判断代码当前最主要的问题是重复、职责混杂还是依赖混乱。
3. 再选择最小重构动作。
4. 只有在收益明确时，才扩大到更高层级的架构调整。

## 对门面类的判断

- 门面类、协调类、ViewModel 往往天然会暴露很多方法，不要只看方法数量。
- 如果大多数方法只是稳定地把命令转发给单一协作对象，这通常比把入口分散到多个小类更容易维护。
- 真正需要优先拆出的，通常是解析规则、格式转换、跨边界协调或重复副作用逻辑。

## 对状态仓库的判断

- 如果派生状态的重建集中在少量核心函数中，这通常说明类的主职责仍然清晰。
- 当很多命令方法都围绕同一份状态做修改时，优先考虑提炼状态变更模板，而不是按页面或字段机械切分类。
