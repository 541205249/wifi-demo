# 边界与架构判断

## 目录

- 架构总原则：模块落点、依赖方向、抽象时机、兼容性与演进、决策顺序
- 入口与页面边界：工作台/容器页、Activity 入口页、demo 首页、设置页、历史详情页、业务台账页、程序页、报告页、设备连接页
- 基础设施边界：前台 Service、导出/分享辅助类、日志辅助类、配置仓库、分发核心、并发通信核心
- 门面与仓库判断：对门面类的判断、对状态仓库的判断

## 模块落点

- 公共模块只放稳定、可复用、跨场景成立的能力，不放业务特例。
- 业务模块优先承载面向场景的流程和规则，不把偶发需求过早下沉为基础设施。
- 如果两个模块都出现相同基础能力，再判断是否值得下沉。

## 依赖方向

- UI 层不要直接依赖底层存储、通信或协议细节，优先通过清晰边界访问。
- 高层策略不应被低层实现细节反向牵制。
- 如果一个类知道太多下层细节，优先考虑收敛接口而不是继续扩展调用链。
- `ViewModel` 适合承接界面交互入口和状态暴露，但不应长期承担协议解析、字符串载荷拆解、持久化格式转换等可独立演化的细节。

## 抽象时机

- 出现一次重复，不急着抽象。
- 出现稳定的重复模式，再抽象。
- 如果抽象后只是多了一层转发而没有降低变化成本，先不抽。

## 兼容性与演进

- 对外协议、存储格式、硬件行为、历史数据兼容优先于结构整齐。
- 重构前先识别真实稳定点，不要在高风险边界上做样式型重写。
- 对线程、socket、硬件通信这类边界，优先保护时序和清理语义，再谈结构优化。
- 如果一段代码同时参与资源清理和监听器通知，把它视为受保护边界；优先在旁路辅助函数上做重构。
- 如果一个前台 `Service` 同时持有通知、唤醒锁、WiFi 锁、socket 和缓存归档状态，默认把它当成高风险边界，不要为了“分层更漂亮”先重排主流程。
- 如果一个基础设施核心同时管理 route、session、fallback 和自动清理，把它当成高风险边界；优先保护分发语义，而不是追求“结构完全统一”。
- 如果一个日志或导出辅助类同时触及缓存目录、压缩文件、系统分享和主线程回调，把它视为资源边界；优先保护资源时序和清理语义。
- 如果一个页面类本身承担“工作台布局骨架”和“交互入口编排”，这通常就是它的合理边界；先保护页面结构可读性，不要急着打散。
- 如果一个 Activity 本身承担“页面入口 + 服务绑定 + 文档选择 + 页面级按钮接线”，这通常也是合理边界；先保护入口流程可读性，再谈拆层。
- 如果一个 demo 入口页本身就是演示台，直接持有 View、日志面板和服务回调也可能是合理边界；优先保护操作流程和生命周期清理。
- 如果一个设置页本身承担“表单控件 <-> 配置模型”的双向映射，这通常也是合理边界；优先保护字段映射的直接可读性。
- 如果一个基础权限委托同时承载提示弹框、申请入口、结果收集和跳系统设置页，这通常也是合理边界；优先保护权限时序，而不是急着拆成多层策略。
- 如果一个日志设置页同时承载表单配置、导出入口和结果提示，这通常也是合理边界；优先保护入口清单和导出注册时机。
- 如果一个配置仓库同时承载持久化和运行时应用，这通常也是高风险边界；优先保护落盘与应用顺序。
- 如果一个轻量 demo `ViewModel` 同时承载少量场景入口、loading 切换和结果发布，这通常也是合理边界；优先保护显式场景入口，不要急着抽成通用场景引擎。
- 如果一个 demo `Fragment` 同时承载动作面板、动态列表、弹框和权限申请，这通常也是合理边界；优先保护显式动作清单，不要急着拆散交互入口。
- 如果一个 demo 首页本身承担“样例目录 + 说明入口”，这通常也是合理边界；优先保护显式导航清单。
- 如果一个 Activity 本身承担“容器页 + 顶部状态区 + 子 Fragment 挂载”，这通常也是合理边界；优先保护容器骨架和挂载时序。
- 如果一个页面本身承担“多个场景按钮 + 一组状态展示 + 说明入口”，这通常也是合理边界；优先保护显式场景清单。
- 如果一个页面本身承担“Tab + ViewPager + 场景分页标题”，这通常也是合理边界；优先保护 tab 顺序和 pager 绑定骨架。
- 如果一个页面本身承担“历史摘要 + 过滤按钮 + 日志正文”，这通常也是合理边界；优先保护过滤规则和摘要结构。
- 如果一个页面本身承担“业务台账列表 + 当前选中摘要 + 弹窗编辑器”，这通常也是合理边界；优先保护动作清单和字段顺序。
- 如果一个页面本身承担“当前程序摘要 + 程序卡片列表 + 步骤卡片列表 + 自定义步骤弹窗”，这通常也是合理边界；优先保护程序顺序、步骤顺序和页面渲染骨架。
- 如果一个页面本身承担“当前报告摘要 + 指标卡片列表 + 历史报告列表 + 二维码入口”，这通常也是合理边界；优先保护报告结构、指标分组和历史展示骨架。
- 如果一个页面本身承担“服务状态 + 在线模块列表 + 已知设备台账 + 日志面板”，这通常也是合理边界；优先保护操作入口、设备摘要和日志展示骨架。
- 如果一个网关接口本身只承担“向上层暴露少量稳定能力”，这通常也是合理边界；优先保护接口直接性，不要为了抽象层次继续套壳。
- 如果一个基础页面基类本身只承担“共享 ViewModel + 少量 UI helper + 页面壳能力”，这通常也是合理边界；优先保护轻量复用骨架。
- 如果一个轻量摘要模型本身只承担“字段承载 + 简单读取”，这通常也是合理边界；优先保护字段直达性，不要急着包装。

## 决策顺序

1. 先确认外部行为和兼容性约束。
2. 再判断代码当前最主要的问题是重复、职责混杂还是依赖混乱。
3. 再选择最小重构动作。
4. 只有在收益明确时，才扩大到更高层级的架构调整。

## 对门面类的判断

- 门面类、协调类、ViewModel 往往天然会暴露很多方法，不要只看方法数量。
- 如果大多数方法只是稳定地把命令转发给单一协作对象，这通常比把入口分散到多个小类更容易维护。
- 真正需要优先拆出的，通常是解析规则、格式转换、跨边界协调或重复副作用逻辑。
- 小型路由分发表、导航映射和命令编排通常可以继续留在协调层，只要决策规则清晰、执行路径短且稳定。
- 领域流程引擎如果主要职责是表达“步骤如何前进/回退/跳过”，这通常是合理的领域边界；不要因为分支多就急着外提为规则引擎。
- 页面级协议处理器注册、路由注册或配置项接线，如果服务于单一页面场景，通常继续留在该页面的 `ViewModel` 或协调层更清楚；先抽接线骨架，不要急着做跨页面配置中心。
- demo / 教学型页面的叙事脚本本身就是边界的一部分；优先保护“一个方法讲清一个场景”的可读性，不要为了配置化把讲解拆散。
- seed data / 初始化素材类的默认内容本身也是边界的一部分；优先保护“默认数据长什么样”的可直接阅读性。
- 命令目录和 Profile 配置类的显式条目列表本身也是边界的一部分；优先保护“系统支持哪些编码/配置位”的可直接阅读性。
- 命令码常量类和微型展示 helper 组合出现时，两者通常对应不同变化原因；优先保护常量清单，只在旁边保留最小的纯文本 helper。
- 设置表单页里的显式字段映射本身也是边界的一部分；优先保护“哪个控件写回哪个字段”的可直接定位性。
- 权限委托里的“何时弹框、何时申请、何时跳设置页”本身也是边界的一部分；优先保护这些时序语义。
- 日志设置页里的显式导出入口本身也是边界的一部分；优先保护“支持哪些导出动作”的可直接阅读性。
- 配置仓库里的“何时落盘、何时 init/apply 到运行时”本身也是边界的一部分；优先保护这些副作用时序。
- 轻量异步场景 ViewModel 里的显式 `runXxxExample()` 入口本身也是边界的一部分；优先保护“支持哪些动作”的可直接阅读性。
- 演示功能面板里的显式按钮入口本身也是边界的一部分；优先保护“当前页面能演示哪些动作”的可直接阅读性。
- demo 首页里的显式导航入口本身也是边界的一部分；优先保护“当前仓库有哪些样例入口”的可直接阅读性。
- 容器页里的 Fragment 挂载顺序和顶部状态映射本身也是边界的一部分；优先保护这些编排语义。
- 场景集合页里的显式按钮入口本身也是边界的一部分；优先保护“当前页面支持哪些场景”的可直接阅读性。
- Tab 场景页里的标题顺序和分页映射本身也是边界的一部分；优先保护这些导航语义。
- 历史详情页里的过滤规则和摘要字段本身也是边界的一部分；优先保护这些展示语义。
- 业务台账页里的动作入口、字段顺序和卡片摘要本身也是边界的一部分；优先保护这些业务展示语义。
- 程序页里的程序顺序、步骤顺序和当前摘要本身也是边界的一部分；优先保护这些流程展示语义。
- 报告页里的摘要结构、指标分组和历史卡片摘要本身也是边界的一部分；优先保护这些报告展示语义。
- 设备连接页里的操作入口、设备卡片摘要和日志面板结构本身也是边界的一部分；优先保护这些运维展示语义。
- 薄网关接口里的方法清单和直接语义本身也是边界的一部分；优先保护这些桥接语义。
- 基础页面基类里的共享 helper 集合和调用方式本身也是边界的一部分；优先保护这些复用语义。
- 轻量摘要模型里的字段结构本身也是边界的一部分；优先保护这些数据表达语义。
- 纯格式化类的格式规则集中性也是边界的一部分；优先保护“展示格式在哪里定义”的可直接定位性。

## 对状态仓库的判断

- 如果派生状态的重建集中在少量核心函数中，这通常说明类的主职责仍然清晰。
- 当很多命令方法都围绕同一份状态做修改时，优先考虑提炼状态变更模板，而不是按页面或字段机械切分类。
- 对持久化仓库，如果重复主要出现在事务内骨架，优先做局部 helper 提炼；不要轻易把事务语义拆散到多个协作层。
