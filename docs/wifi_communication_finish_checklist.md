# WiFi 通信框架收尾清单

## 1. 当前底座完成度

当前项目里的 WiFi 通信底座已经具备下面这些能力：

- TCP 长连接服务
- 设备连接 / 断开记录
- 通信记录落库
- MAC 识别归档
- 编码表加载与校验
- `command` 普通指令层
- `ack` 统一回执层
- `transfer` 分片传输层
- `stream` 实时流层
- `gateway` 协议识别层
- `dispatcher` 业务分发层

可以认为“底座框架”已经完成，下一阶段重点不再是搭框架，而是联调、约束补齐、正式业务接入。

## 2. 联调前必须确认的事项

下面这些内容要在正式和设备端联调前先定好：

### 2.1 设备身份

- 模块是否一定支持查询 MAC
- MAC 查询命令是否已固定
- MAC 返回格式是否固定
- 模块是否会出现更换主控但保留历史档案的情况

### 2.2 连接规则

- 模块主动连 App，还是 App 主动找模块
- 同时允许几个模块连接
- 断线后的重连策略是什么
- 心跳周期是多少
- 多久无响应算掉线

### 2.3 指令规则

- 每个业务动作是否都有独立编码
- 请求和确认回包是否分开编号
- 错误回包是否统一走 `ACK+ / ERR+`
- 动态返回字段是否统一前缀或正则格式

### 2.4 传输规则

- 文件传输最大大小
- 分片大小上限
- 是否允许断点续传
- 是否允许重发单片
- 是否要加总包校验

### 2.5 实时流规则

- 推流开始和结束指令
- 每秒帧数范围
- 单帧最大长度
- 丢帧是否允许
- 遇到丢帧是继续还是重建会话

## 3. 设备联调阶段待办

### 3.1 第一阶段：先打通握手与连接

- 用真实模块接入 `TcpServerService`
- 确认服务端口 `39509` 不变
- 验证模块可正常连接、断开、重连
- 验证连接记录和断开记录时间戳是否正确
- 验证 MAC 识别是否能归档到同一设备

### 3.2 第二阶段：打通编码表机制

- 让设备端拿模板 CSV 填写真实命令
- 在手机端通过命令设置页加载 CSV
- 检查缺失编码、未知编码、未填命令是否提示正确
- 确认发送编码可正确映射到真实指令
- 确认接收消息可正确反查到编码

### 3.3 第三阶段：打通普通命令

- 查询模块信息
- 切换自动 / 手动模式
- 开始验光
- 停止验光
- 设备状态上报
- 验光结果上报

这一阶段要重点确认：

- 请求与回包是否都命中对应编码
- ACK 是否稳定
- dispatcher 是否能正确进入业务处理器

### 3.4 第四阶段：打通大消息能力

- 模板文件发送
- 模块日志包接收
- 波形流 / 连续测量值推送

这一阶段要重点确认：

- transfer 分片顺序
- 分片 CRC 校验
- 最终组包 MD5
- stream 丢帧统计
- 会话终态自动移除是否正常

## 4. 正式业务接入待办

框架完成后，正式业务还要继续接下面这些内容：

- 把各页面按钮全部接到固定编码常量
- 把收到的结果写入 Room 数据库
- 把验光流程状态机接到 dispatcher
- 把异常提示、失败重试、超时处理接上
- 把历史记录页接到真实结果数据

建议正式业务层按下面方式接：

- 页面只负责交互和展示
- ViewModel 负责状态流转
- Repository 负责命令发送、结果持久化、网络 / 本地组合数据
- 通信 Service 只负责连接和消息通道

## 5. 还建议补的工程能力

这些不属于“通信框架是否成立”的范围，但属于真正上线前很有必要补的内容：

- 指令发送超时机制
- ACK 超时机制
- 自动重试策略
- 会话取消机制
- 文件传输失败补偿
- 流会话超时停止
- 指令版本兼容策略
- 编码表版本标识与回滚策略

## 6. 上线前建议做的稳定性验证

### 6.1 连接稳定性

- 连续运行 2 到 4 小时不断连
- 模块反复断开 / 重连
- 手机切后台、锁屏、息屏
- WiFi 切换或弱网波动

### 6.2 通信稳定性

- 高频连发普通指令
- 大文件连续上传 / 下载
- 长时间实时流接收
- 多种协议消息混发

### 6.3 数据正确性

- 日志记录完整
- 时间戳正确
- 历史设备归档不串设备
- 结果字段解析无错位
- CSV 变更后映射仍正确

## 7. 建议的优先级

建议下一阶段按这个顺序推进：

1. 先和设备端把真实 CSV 编码表填完整
2. 用真实模块打通握手、开始、停止、状态、结果
3. 补 ACK / 超时 / 重试约束
4. 再接正式验光业务页面
5. 最后做文件传输、实时流和长稳测试

## 8. 当前结论

如果只问“框架底座是否完成”，答案是：已经完成。

如果问“是否已经达到可直接量产交付的程度”，答案是：还差真实协议联调、超时重试、正式业务状态机和长稳验证。
