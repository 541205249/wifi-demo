# WiFi 模块常见通信场景设计说明

## 1. 目的

这份文档说明 App 和 WiFi 模块之间常见的通信类型，以及在当前 `lib.command` 框架下建议怎么设计。

当前这套 `command` 包最适合做的是：

- 小指令控制
- 查询与应答
- 状态上报
- 结果上报
- 少量字段传递
- 心跳、错误码、确认回包

当前这套 `command` 包不适合单独承担的是：

- 大文件上传下载
- 固件升级
- 长连接实时流数据
- 大量二进制分片传输

这些场景建议在 `command` 之上再加一层单独的传输层，例如未来新增 `transfer` 子包。

## 2. 先分三类

一般 App 和 WiFi 模块通信，基本可以分成三类：

| 类别 | 特点 | 典型场景 |
| --- | --- | --- |
| 控制类 | 消息短，要求明确执行某个动作 | 开关灯、开始验光、停止、切模式 |
| 数据类 | 消息里带几个字段或结果值 | 设置参数、上报状态、上报验光结果 |
| 传输类 | 消息体大，往往要分片和校验 | 发文件、收文件、升级包、日志包 |

## 3. 当前编码框架怎么用

当前项目编码规则是：

- `s` 开头表示 App 发给模块
- `r` 开头表示模块发给 App
- 一个编码只表示一个语义
- 请求和确认回包最好拆成两个编码

编码表文档当前只保留三列：

| 列名 | 说明 |
| --- | --- |
| 编码 | 唯一编码，例如 `s120101` |
| 指令 | 真实命令内容 |
| 编号解释 | 说明这个编码是干什么的 |

`指令` 这一列的规则：

- 普通文本：默认按精确匹配
- `prefix:` 开头：按前缀匹配
- `regex:` 开头：按正则匹配

例子：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s100101` | `AT+INFO?` | 查询模块信息 |
| `r100102` | `regex:^INFO\\+(.+)$` | 模块信息上报 |
| `r120201` | `prefix:STATUS+` | 设备状态上报 |

## 4. 常见通信场景

### 4.1 单动作控制

这是最常见的场景，特点是：

- 消息短
- 参数少
- 一个按钮对应一个动作
- 通常需要一个确认回包

典型例子：

- 开灯 / 关灯
- 开始 / 停止
- 切自动 / 切手动
- 打印 / 停止打印

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s210101` | `AT+LAMP=ON` | 打开补光灯 |
| `r210102` | `AT+LAMP=ON` | 补光灯打开确认 |
| `s210103` | `AT+LAMP=OFF` | 关闭补光灯 |
| `r210104` | `AT+LAMP=OFF` | 补光灯关闭确认 |

建议：

- 一个发送动作配一个接收确认
- 不要把“请求”和“确认”混成一个编码
- 即使请求字符串和确认字符串暂时一样，编码也要分开

### 4.2 带几个字段的设置

这是第二常见的场景，特点是：

- 需要带 2 到 10 个字段
- 一般是设置参数或提交一次业务数据

典型例子：

- 设置球镜、柱镜、轴位
- 设置程序参数
- 设置时间、阈值、模式

建议优先使用“固定格式文本 + 占位符”。

示例一：键值对风格

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s220101` | `SET+REF=SPH:${sph},CYL:${cyl},AXIS:${axis}` | 设置验光参数 |
| `r220102` | `AT+REF=OK` | 验光参数设置确认 |

示例二：如果字段稍多，也可以写成 JSON 字符串

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s220201` | `{"cmd":"set_ref","sph":"${sph}","cyl":"${cyl}","axis":"${axis}"}` | 设置验光参数(JSON) |

建议：

- 如果模块端资源有限，优先用固定文本格式，不要一上来就 JSON
- 字段顺序尽量固定
- 字段名尽量短，但要能看懂
- 所有可变字段都通过占位符 `${name}` 注入

### 4.3 查询与应答

这是 App 主动问，模块返回结果。

典型例子：

- 查询模块信息
- 查询版本号
- 查询当前模式
- 查询设备状态

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s230101` | `AT+VER?` | 查询模块版本 |
| `r230102` | `regex:^VER\\+(.+)$` | 模块版本返回 |
| `s230103` | `AT+MODE?` | 查询当前模式 |
| `r230104` | `regex:^MODE\\+(.+)$` | 当前模式返回 |

建议：

- 查询和查询结果分两个编码
- 接收值不固定时用 `regex:`
- 返回内容有固定前缀时优先用 `prefix:` 或 `regex:`

### 4.4 状态上报

这是模块主动往 App 推状态。

典型例子：

- 设备空闲
- 正在验光
- 验光暂停
- 设备故障
- 纸仓缺纸
- 电机忙碌

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `r240101` | `prefix:STATUS+` | 设备状态上报 |
| `r240102` | `prefix:WARN+` | 设备告警上报 |
| `r240103` | `prefix:ERROR+` | 设备错误上报 |

建议：

- 状态、告警、错误尽量拆开
- 状态值如果比较固定，可以再细分编码
- 如果状态种类特别多，也可以先统一前缀，再在业务层解析内容

### 4.5 结果上报

这是模块把一次业务结果回传给 App。

典型例子：

- 验光结果
- 角膜曲率结果
- 视力检查结果
- 打印结果

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `r250101` | `regex:^RESULT\\+SPH=([^,]+),CYL=([^,]+),AXIS=([^,]+)$` | 验光结果上报 |
| `r250102` | `regex:^PRINT\\+RESULT=(OK|FAIL)$` | 打印结果上报 |

建议：

- 少量固定字段用正则就够
- 如果字段很多，但仍然不大，可以考虑 JSON 文本
- 如果一条结果会超过普通一行文本很多，就要考虑分包

### 4.6 心跳与在线保持

这是连接层非常常见的场景。

典型例子：

- App 定时发心跳
- 模块回心跳
- 模块定时主动报活跃

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s260101` | `PING` | App 心跳 |
| `r260102` | `PONG` | 模块心跳回包 |
| `r260103` | `prefix:ALIVE+` | 模块主动保活上报 |

建议：

- 心跳尽量简单
- 不要把业务数据塞进心跳
- 心跳失败后由连接层决定重连，不要交给业务页面处理

### 4.7 错误、拒绝、失败原因

这个场景经常被忽略，但非常重要。

典型例子：

- 参数非法
- 当前状态不允许执行
- 模块忙
- 权限不足
- 文件校验失败

建议写法：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `r270101` | `prefix:ERR+` | 通用错误回包 |
| `r270102` | `regex:^ERR\\+CODE=([^,]+),MSG=(.+)$` | 错误码与错误信息 |

建议：

- 统一保留一套通用错误回包编码
- 尽量有错误码，不只返回 `FAIL`
- 有错误码后，App 更容易做提示和重试策略

### 4.8 批量小数据

这是介于“字段消息”和“文件传输”之间的场景。

典型例子：

- 一次上传 20 条小记录
- 一次下发一组参数
- 一次下发一个检查流程模板

建议写法：

- 如果总长度不大，可以直接文本化
- 如果长度已经接近 TCP 一次业务包上限，就不要继续塞在一条命令里

示例：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s280101` | `DATA+LIST=${payload}` | 批量小数据下发 |
| `r280102` | `AT+LIST=OK` | 批量小数据接收确认 |

建议：

- 小数据才这样做
- 真正的“大内容”不要只靠 `${payload}` 一把梭

## 5. 文件传输怎么设计

文件传输和普通命令不一样，建议拆成两层：

- 命令层：只负责“开始传文件、准备好、完成、失败”
- 传输层：真正负责分片、重传、校验、进度

换句话说：

- `CommandEngine` 适合管控制命令
- 文件块本身不建议直接全塞进 `CommandEngine`

### 5.1 文件上传

App 发文件给模块时，建议流程如下：

1. App 发送“开始上传”命令
2. 模块返回“已准备好”与会话 ID
3. App 按 chunk 分片发送二进制或 Base64 数据
4. 模块按片确认
5. 全部完成后模块返回“文件完成”

建议命令层这样定义：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s310101` | `FILE_BEGIN+TYPE=${type},NAME=${name},SIZE=${size},MD5=${md5}` | 开始上传文件 |
| `r310102` | `regex:^FILE_READY\\+SESSION=([^,]+),CHUNK=([^,]+)$` | 模块准备好接收文件 |
| `r310103` | `prefix:FILE_PROGRESS+` | 文件接收进度 |
| `r310104` | `FILE_DONE` | 文件接收完成 |
| `r310105` | `prefix:FILE_ERR+` | 文件接收失败 |

建议：

- 真正的数据块不要走编码表
- 数据块建议单独定义 chunk frame
- 每片最好带 `sessionId + index + checksum`

### 5.2 文件下载

模块发文件给 App 时，建议流程相反：

1. App 发下载请求
2. 模块返回文件元信息
3. 模块开始按片发送
4. App 逐片确认
5. 模块发送完成

建议命令层这样定义：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s320101` | `FILE_GET+TYPE=${type}` | 请求下载文件 |
| `r320102` | `regex:^FILE_META\\+NAME=([^,]+),SIZE=([^,]+),MD5=([^,]+)$` | 文件元信息返回 |
| `r320103` | `prefix:FILE_BEGIN+` | 开始发送文件 |
| `r320104` | `FILE_DONE` | 文件发送完成 |
| `r320105` | `prefix:FILE_ERR+` | 文件发送失败 |

## 6. 实时流数据怎么设计

如果模块会持续不断上报数据，比如：

- 实时波形
- 实时测量值
- 传感器持续输出

不建议每一帧都走“一个编码匹配一次业务逻辑”的模式。

建议拆成三层：

- `command` 层：控制开始、停止、流状态
- `stream` 层：实际流数据
- `ui` 层：订阅流并刷新界面

建议命令层这样定义：

| 编码 | 指令 | 编号解释 |
| --- | --- | --- |
| `s330101` | `STREAM_START+TYPE=${type}` | 开始实时流 |
| `r330102` | `STREAM_START=OK` | 实时流开始确认 |
| `s330103` | `STREAM_STOP+TYPE=${type}` | 停止实时流 |
| `r330104` | `STREAM_STOP=OK` | 实时流停止确认 |
| `r330105` | `prefix:STREAM_ERR+` | 实时流错误 |

真正的每帧数据：

- 如果很小，可以额外定义统一前缀，例如 `prefix:FRAME+`
- 如果频率高，建议二进制帧，不要只靠字符串匹配

## 7. 推荐写法总结

### 7.1 最推荐的规则

- 一个编码只表示一个业务语义
- 请求和回包分开编码
- 小命令优先用简单文本
- 多字段优先用固定键值对
- 动态接收值用 `regex:`
- 固定头部上报用 `prefix:`
- 大文件和实时流不要直接塞进普通命令层

### 7.2 什么时候适合继续用当前 `command` 包

- 点按钮发一个动作
- 查询一个状态
- 收一个确认
- 收几个字段的结果
- 收一个错误码
- 做基础联调

### 7.3 什么时候应该新建更高一层能力

- 文件上传下载
- 升级包传输
- 大量图片、日志、报告传输
- 高频实时流
- 需要断点续传、重试、校验的大块数据

## 8. 对你这个项目的直接建议

结合当前验光仪项目，建议优先把通信分成下面几类模块：

| 模块 | 通信内容 |
| --- | --- |
| 系统握手 | 查询模块信息、查询版本、能力协商 |
| 流程控制 | 开始、停止、暂停、恢复、切模式 |
| 参数设置 | 验光参数、程序参数、显示参数 |
| 状态上报 | 空闲、忙碌、故障、告警 |
| 结果上报 | 验光结果、打印结果、检查结果 |
| 心跳保活 | 连接存活、断线检测 |
| 文件传输 | 报告模板、日志包、升级包 |
| 实时流 | 波形、连续测量值 |

其中：

- 前六类直接适合当前 `lib.command`
- 后两类建议未来单独做 `transfer` 或 `stream` 层

## 9. 结论

最简单的一句话是：

- 小消息，用 `command`
- 大消息，用 `command + transfer`
- 高频流，用 `command + stream`

当前项目里这三层已经对应到：

- 普通命令：`lib.command`
- 文件传输：`lib.command.transfer`
- 实时流：`lib.command.stream`
- 统一回执：`lib.command.ack`
- 业务分发：`lib.command.dispatcher`

其中统一 ACK / 错误码模型如果你后面还要，我可以再继续补。

## 12. 为什么还要单独做 dispatcher 层

很多项目在把协议识别做出来之后，业务层还是会继续写很多散落判断：

- `if code == r120202`
- `if ack.ref == s120101`
- `if sessionId == file001`

这样时间一长，页面和 ViewModel 会重新和协议细节耦合起来。

现在项目里又补了 `lib.command.dispatcher`，建议这样理解：

- `gateway` 负责把原始消息识别成协议事件
- `dispatcher` 负责把协议事件路由到业务处理器
- `viewmodel / repository` 只处理业务动作

对应关系可以简单记成：

| 层 | 负责什么 |
| --- | --- |
| `command / ack / transfer / stream` | 定义协议模型 |
| `gateway` | 识别原始消息属于哪一类 |
| `dispatcher` | 按编码、ACK 引用、sessionId 路由到业务处理器 |
| 业务层 | 刷 UI、存库、推进流程 |

这样后面正式做验光流程时：

- 验光结果上报可以按 `command code` 分发
- 开始/停止回执可以按 `ACK ref` 分发
- 模板文件可以按 `transfer sessionId` 分发
- 波形流可以按 `stream sessionId` 分发

## 11. 为什么还要单独做 ACK 层

很多项目一开始只关注“怎么发命令”，后面联调时才发现更麻烦的是“模块怎么回”。

如果没有统一回执层，常见情况会变成：

- 普通命令回一套格式
- 文件传输回另一套格式
- 实时流错误又是第三套格式

这样 App 端就得写很多分散的判断逻辑。

现在项目里单独补了 `lib.command.ack`，建议你把：

- 普通命令成功 / 失败
- 文件传输成功 / 失败
- 实时流开始 / 中断 / 停止

尽量都统一回成：

- `ACK+...`
- `ERR+...`

这样 App 拿到任何一条回执，先统一解析成：

- `status`
- `type`
- `ref`
- `session`
- `errorCode`

再决定后面的业务处理。

## 10. 用业务话翻译这些场景

如果不用协议术语，而是直接用业务开发能听懂的话，可以这样理解：

| 通信场景 | 它到底是什么意思 | 一个直白例子 |
| --- | --- | --- |
| 单动作控制 | App 想让模块立刻做一件事 | 点“开始验光”，App 发 `START+OPT`，模块开始执行 |
| 带字段设置 | App 要把几个参数一起交给模块 | 设球镜、柱镜、轴位，模块按这些参数工作 |
| 查询与应答 | App 不知道当前值，所以主动去问 | App 问“你当前版本是多少”，模块回 `VER+1.0.8` |
| 状态上报 | 模块主动告诉 App 自己现在是什么状态 | 模块自己推 `STATUS+BUSY`，App 立刻把按钮置灰 |
| 结果上报 | 模块把真正的业务结果回给 App | 模块发 `RESULT+SPH=-1.25,CYL=-0.50,AXIS=180` |
| 文件传输 | 一条命令装不下，需要分片慢慢传 | App 发模板文件、升级包、日志包给模块 |
| 实时流 | 数据持续不断，不是一问一答 | 模块连续发波形或实时测量值 |

### 10.1 单动作控制

你可以把它理解成“遥控器按一下，设备做一下”。

例如：

- 用户点“打开补光灯”
- App 根据编码拿到真实指令 `AT+LAMP=ON`
- 模块执行后回 `ACK+LAMP`
- App 把补光灯状态改成“已开启”

它最适合：

- 开关
- 开始 / 停止
- 切模式
- 单次触发动作

### 10.2 带字段设置

你可以把它理解成“把一张表单交给模块”。

例如：

- 页面上填写 `SPH=-1.25`
- 页面上填写 `CYL=-0.50`
- 页面上填写 `AXIS=180`
- App 拼成一条设置消息发给模块
- 模块确认后，App 提示“参数已同步”

它最适合：

- 验光参数设置
- 程序阈值设置
- 时间、模式、开关组合配置

### 10.3 查询与应答

你可以把它理解成“App 问一句，模块答一句”。

例如：

- App 刚连上模块，想知道固件版本
- App 发 `AT+VER?`
- 模块回 `VER+1.0.8`
- App 在页面上显示版本号

它最适合：

- 查询版本
- 查询当前模式
- 查询设备能力
- 查询某项配置的当前值

### 10.4 状态上报

你可以把它理解成“模块在主动报平安、报进度、报异常”。

例如：

- 模块空闲时发 `STATUS+READY`
- 开始执行任务后发 `STATUS+BUSY`
- 异常时发 `ERROR+MOTOR`
- App 根据状态更新按钮、状态灯和提示语

它最适合：

- 空闲 / 忙碌 / 暂停
- 故障 / 告警
- 流程阶段变化

### 10.5 结果上报

你可以把它理解成“模块把最终成绩单交回来”。

例如：

- 模块完成一次验光
- 发 `RESULT+SPH=-1.25,CYL=-0.50,AXIS=180`
- App 解析后刷新结果卡片
- 同时可以存库、生成报告、供历史页查看

它最适合：

- 验光结果
- 打印结果
- 检查结果
- 单次测量完成后的摘要数据

### 10.6 文件传输

你可以把它理解成“快递太大，不能一口气塞进一条消息里，只能先打包、分箱、逐箱签收”。

例如：

1. App 先说：“我要发一个模板文件”
2. 模块回：“可以，按 2048 字节一片发，sessionId 是 xxx”
3. App 一片一片发
4. 模块一片一片收
5. 全部收完后模块回 `FILE_DONE`

它最适合：

- 模板文件
- 日志包
- 升级包
- 报告文件

### 10.7 实时流

你可以把它理解成“像水龙头一样持续往外流数据”，不是“发一条回一条”的模式。

例如：

- 模块持续输出波形
- 模块每 50ms 上报一次实时测量值
- App 不断刷新曲线图

它最适合：

- 波形
- 连续测量值
- 高频传感器数据

所以对你这个项目，最实用的一句判断标准就是：

- 只做一个动作：单动作控制
- 传几个参数：带字段设置
- 想知道当前值：查询与应答
- 模块主动报状态：状态上报
- 模块回业务结果：结果上报
- 一条装不下的大内容：文件传输
- 持续不断的数据：实时流
